# Сверка: Cloud транзакции vs OLAP (iikoServer)

Цель — по чек-листу MVP: «Выручка за день / заказы / скидки / оплаты / доставка сходятся с iiko».

## Уровень агрегации сверки (минимум)

Общий ключ:

- календарный **`business_date`** (ясно объявите таймзону точки продаж или UTC в правилах);
- **`organization_id`** (или `department_code`, если OLAP режется по нему без UUID);
- при необходимости **`order_source`** / канал, если офисный отчёт режет по источнику.

## Что брать источником истины

| Разрез | Главная копия | Вторая (контроль) |
|--------|---------------|-------------------|
| Количества заказов, позиций чека, суммы строк, оплат, промо в разрезе гостевого счёта | **Cloud транзакции** (полные документы или выгрузка сущности «чек», когда покрыты методами лицензии) | **OLAP SALES**/аналог только для агрегатной сверки |
| Выручка «как в офисном отчёте» после проводок OLAP-пресета | Использовать **OLAP как эталон** для сверки `mart_daily_sales` только если офис утверждает этот пресет | При расхождениями — трассировать на уровень **позиций**/чеков в Cloud |

Правило большого пальца: **тонкая модель для BI и гранулярных дашбордов — Cloud/`fact_*`; OLAP для «совпало с главной книгой» и случаев, где склад/себестоимость задаётся привычными отчётами Chain.**

## Допуски

- Числовой epsilon: суммы округляются на границе фискальных копеек — типичный допуск \(\le\) **0.01 × число платежных сплитов**.
- По времени: **business_date** в Cloud может сдвигаться от календаря «офисной даты закрытия» — описать процедуру «дата счёта = дата события X» один раз и не смешивать.

## KPI сверочного мониторинга

1. По каждой паре источников: `Δ = sum_cloud_net_revenue − sum_olap_net_revenue`.
2. `|Δ| / gross` > порога (например 1%) поднимает алерт.
3. По доставке: сравниваются **counts** закрытых доставок по дню и сумма времени SLA (если OLAP поддерживает).

## Связка с кодом синка

- В сырой слой сохранять `source_system`: `cloud` | `server_olap`.
- Прогоны reconciliation — отдельный SQL/materialized view: `mart_reconcile_daily` (опционально в следующих итерациях после стабильного ingestion).
